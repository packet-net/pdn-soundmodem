# Reference samples

Two sets, kept for different reasons.

| directory | what | reproducible? |
|---|---|---|
| `pdn/` | our transmission, one WAV per mode | **yes** — `sm-samples` regenerates them byte-for-byte |
| `ninotnc/` | a real NinoTNC's transmission, one WAV per mode | **no** — recordings of physical hardware |

`ninotnc/` is the valuable half. Those are recordings of a NinoTNC (firmware 3.44) off pin 1
over the CM108 bench loop (docs/ninotnc-loop.md), and they are the source of the per-mode
occupied-bandwidth references that `OccupiedBandwidthTests` holds us to — the rule being
that we are never wider than the TNC we share a channel with. They cannot be regenerated
without that rig, that TNC and that firmware, so they are checked in. Level is as captured
(~0.17 peak, AGC off, fixed +8 dB mic gain); normalise before use if you need a known level.

**Caveat on those references** — see issue #2: they are measured over a different window
from our own signal (their bursts are mostly preamble), so the comparison in the test table
is not yet like-for-like. The recordings are sound; the derived numbers need re-deriving.

`pdn/` is regenerable and is checked in only so the modes can be listened to, played into
other hardware, or diffed after a modulator change without building anything. Expect to
regenerate rather than trust these once #1/#3/#4 land — and prefer replacing them over
accumulating variants, since each set is ~32 MB of binary in history.

# pdn/ — one WAV per NinoTNC mode

Generated by `tools/Packet.SoundModem.Samples` (`sm-samples`), 2026-07-15.

## Format

- **Mono, 48 kHz, 16-bit** — the card rate we actually transmit at, rendered through the
  same upsampler the daemon uses, so these are what goes on the wire.
- **10 UI frames** per file, payloads of **10, 30, 50, 70, 90, 110, 130, 150, 175, 200**
  bytes, in that order.
- **`Q0AAA > TEST`**, UI/no-layer-3. Payload is readable ASCII, self-describing:
  `PDN <mode> #NN len=LLL ` then A-Z filler.
- **50 ms TXDELAY** on every frame, **1 s of silence** between frames (and 0.5 s of lead-in).

## Verification — and an important caveat

Played into a real NinoTNC (firmware 3.44) over the CM108 bench loop, with the TNC's mode
verified from `BrdSwchMod` first, it decodes these files:

| file | NinoTNC | our own receiver |
|---|---|---|
| mode 12 afsk300 | **9/10** | **0/10** |
| mode 06 afsk1200 | **10/10** | 6/10 |
| mode 04 fsk4800-il2p | **10/10** | 4/10 |
| mode 00, 05, 07, 08, 09, 10, 11, 13, 14 | (spot-checked) | 10/10 |

**The files are correct** — a real TNC decodes them. But at 50 ms TXDELAY **our own
receiver cannot reliably decode our own transmissions**: 300 baud AX.25 gets just two HDLC
flags (16 bits) of preamble, and our slicer needs ~42 bits at that rate to settle. The
NinoTNC's receiver acquires far faster than ours.

So do not read a low "our own receiver" number as a bad WAV. It is our acquisition, tracked
in pdn-soundmodem issue #4.

## Regenerating

```sh
dotnet run --project tools/Packet.SoundModem.Samples -- <outdir> \
    [--txdelay 50] [--gap-ms 1000] [--source Q0AAA] [--dest TEST]
```
