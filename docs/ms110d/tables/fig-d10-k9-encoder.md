# Figure D-10 — Constraint length 9, rate 1/2 convolutional encoder

Source: MIL-STD-188-110D Appendix D, D.5.3.2.2 + Figure D-10, document page 220
(PDF page 225). Image page (no text layer); transcribed by eye at 300 dpi, tap
arrows traced on a 2x-upscaled crop.

## Polynomials as printed (heading "Polynomials:")

```
(b0) T1 = x^8 + x^6 + x^5 + x^4 + 1
(b1) T2 = x^8 + x^7 + x^6 + x^5 + x^3 + x^1 + 1
```

## Prose (D.5.3.2.2, transcribed from image)

> A constraint length 9, rate 1/2 convolutional code shall be used prior to
> puncturing. Figure D-10 is a pictorial representation of the encoder.

Below the figure:

> The two summing nodes in the figure represent modulo 2 additions. For each bit
> input to the encoder, two bits are taken from the encoder, with the upper
> output bit b0, generated by polynomial T1(x), taken first.

## Figure wiring, tap by tap

- Shift register of 9 cells, drawn left to right and labeled
  `X^8, X^7, X^6, X^5, X^4, X^3, X^2, X, 1`.
- The **Input arrow enters the leftmost cell (`X^8`)**; `X^8` holds the newest
  input bit, `1` the oldest (eight shifts ago).
- **Upper adder → b0 (T1)**: taps from cells `X^8, X^6, X^5, X^4, 1`
  (five arrows; matches the printed T1 exactly).
- **Lower adder → b1 (T2)**: taps from cells `X^8, X^7, X^6, X^5, X^3, X, 1`
  (seven arrows; matches the printed T2 exactly).
- Both adders are modulo-2. Output order per input bit: **b0 first, then b1**.

## Codable form (delay-operator convention)

With u[n] the current input bit (cell `X^8`) and u[n-8] the oldest (cell `1`):

```
b0[n] = u[n] ^ u[n-2] ^ u[n-3] ^ u[n-4] ^ u[n-8]                       ->  1 + D^2 + D^3 + D^4 + D^8            =  101110001b  =  0o561
b1[n] = u[n] ^ u[n-1] ^ u[n-2] ^ u[n-3] ^ u[n-5] ^ u[n-7] ^ u[n-8]     ->  1 + D + D^2 + D^3 + D^5 + D^7 + D^8  =  111101011b  =  0o753
```

i.e. **(561, 753) octal**, T1 = 561 emitted first.

## Adjacent context on the same page: D.5.3.2.3 Full-tail-biting encoding (image, transcribed)

> To begin encoding each block of input data, the encoder shall be preloaded by
> shifting in the first (k-1) input data bits without taking any output bits.
> These (k-1) input bits shall be temporarily saved so that they can be used to
> "flush" the encoder. The first two coded output bits shall be taken after the
> (kth) bit has been shifted in, and shall be defined to be the first two bits
> of the resulting block code. After the last input data bit has been encoded,
> the first (k-1) "saved" data bits shall be encoded. Note that the encoder
> shift register should not be changed before encoding these saved bits; i.e.,
> it should be filled with the last seven input data bits. The (k-1) "saved"
> data bits are encoded by shifting them into the encoder one at a time,
> beginning with the earliest of the (k-1). The encoding thus continues by
> taking the two resulting coded output bits as each of the saved (k-1) bits is
> shifted in. These encoded bits shall be the final bits of the resulting
> (unpunctured) block code. Prior to puncturing, the resulting block code will
> have exactly twice as many bits as the input information bits. Puncturing of
> the rate 1/2 code to the other required rates shall be done prior to sending
> bits to the interleaver.

Spec oddity, recorded not resolved: the sentence "filled with the last **seven**
input data bits" hard-codes the K=7 value of (k-1) even though the section is
generic over both codes (for K=9, k-1 = 8; every other reference in the
paragraph says "(k-1)").
